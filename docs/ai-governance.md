# AI 协作治理说明

## 角色关系

- 老板：只提出业务目标、体验偏好、底线和不满意点，不参与技术实现。
- ChatGPT：经理位，负责需求澄清、任务拆解、PR 审核、返工决策和对老板的阶段汇报。
- Codex：执行位，负责代码修改、测试、修复、PR 更新和交付包补齐。
- GitHub：唯一真相源，承载 issue、分支、PR、评论、checks、review 和审计痕迹。
- MCP：ChatGPT 读取 GitHub 现场、仓库文件和检查结果的通道。

## 标准工作流

1. 老板向 ChatGPT 描述业务目标、风格偏好、底线和不满意点。
2. ChatGPT 将自然语言转成结构化任务单：目标、范围、约束、验收标准、风险、给 Codex 的执行指令。
3. Codex 在仓库中创建或更新分支，修改代码，运行 `lint` / `build` / `test` / `typecheck`，并提交 PR。
4. GitHub Actions 产出真实 checks；PR 模板沉淀变更原因、风险和验证方式。
5. ChatGPT 通过 MCP 读取 PR、diff、changed files、checks、测试摘要和关键文件，按 `AGENTS.md` 执行审核。
6. 若不通过，ChatGPT 生成结构化返工单，并回写到 PR 评论；若通过，只向老板汇报效果、风险和上线建议。

## 审核流程

- 审核输入只能来自 GitHub 与仓库现场，不接受“人工转述已经测过”。
- ChatGPT 审核时至少读取：
  - PR 标题、描述、作者、分支、状态
  - 变更文件与 diff 摘要
  - `AGENTS.md`
  - 关键源码或文档文件
  - checks / CI 结果
  - 测试摘要
- 审核输出固定覆盖：
  - 本轮完成内容
  - 未完成内容
  - 发现的问题
  - 风险等级
  - 返工指令
  - 是否建议合并

## 返工流程

1. ChatGPT 发现阻断项后，不直接给老板技术细节。
2. ChatGPT 在 PR 评论中回写结构化返工任务，明确 evidence、影响、修复要求、重新验证要求。
3. Codex 仅根据 PR 评论、PR diff 和仓库规则继续执行下一轮。
4. 新一轮提交后，ChatGPT 重新基于真实 diff 和 checks 审核，不沿用上一轮口头结论。

## 故障处理流程

- MCP 不可用：ChatGPT 只能给出“连接受阻，无法基于真实现场审核”的状态，禁止凭记忆给出通过结论。
- GitHub API 失败：先检查 token、仓库权限、rate limit，再检查 workflow 是否产出了对应 checks。
- CI 未返回：优先判定为“证据不足”，要求补齐 checks 或重新运行。
- Codex workflow 失败：保留失败日志，ChatGPT 基于日志与 PR 现场回写明确返工要求。

## 权限边界

- Boss 不需要仓库写权限来理解技术细节。
- ChatGPT 默认只通过 MCP 读取仓库与 GitHub；写操作只允许最小必要集，例如回写 PR 评论。
- Codex 只能在仓库、PR 和 workflow 授权范围内执行，不应访问无关仓库或额外系统。
- MCP 默认只读；开启写能力时必须显式配置并限制到评论等低风险动作。

## 失败时的降级策略

- 无法连到 MCP：降级为“只给出待连接提示”，不做正式审核结论。
- 无法获取 checks：降级为“证据不足，不能建议合并”。
- 仅有文档改动、无自动化验证：允许通过手工验证，但必须在 PR 和交付包中明确说明。
- GitHub 不可用但本地仓库可读：可先生成任务单和实现草案，但不能把它当成正式审核结果。
